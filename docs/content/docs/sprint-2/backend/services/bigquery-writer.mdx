## DecisĂµes de design

### Por que Go e nĂŁo Python/Node?

O BigQuery e o Kafka tĂŞm SDKs sĂłlidos em Go, a linguagem gera binĂˇrios estĂˇticos sem runtime externo (importante para imagens alpine pequenas) e o modelo de goroutines se encaixa bem no loop de consumer. O time jĂˇ usava Go no `redis-writer`, entĂŁo a escolha manteve consistĂŞncia.

### `segmentio/kafka-go` em vez de `confluent-kafka-go`

A lib da Confluent exige CGO e `librdkafka` compilada no sistema, o que complica o Dockerfile (precisa de gcc, headers C, imagem maior). O `segmentio/kafka-go` Ă© puro Go: `CGO_ENABLED=0` e binĂˇrio estĂˇtico funcionam em `alpine:3.19` sem dependĂŞncias extras.

### Streaming inserts em vez de batch/load jobs

BigQuery suporta dois modos de escrita: streaming inserts (linha a linha, em tempo real) e load jobs (arquivos em batch). Streaming inserts tĂŞm latĂŞncia de segundos e sĂŁo a escolha natural para um consumer Kafka que precisa que os dados estejam disponĂ­veis no Looker logo apĂłs a ingestĂŁo.

### IDs determinĂ­sticos (UUID v5) e deduplicaĂ§ĂŁo

Cada linha recebe um ID derivado das coordenadas da mensagem Kafka via UUID v5 (SHA-1):

```

id(age)          = UUID_v5("geodata:0:42:age")
id(gender)       = UUID_v5("geodata:0:42:gender")
id(social_class) = UUID_v5("geodata:0:42:social_class")
id(target)       = UUID_v5("geodata:0:42:target")

```

O mesmo ID Ă© enviado como `insertId` no streaming insert. O BigQuery usa esse campo para deduplicar retries dentro de uma janela de ~1 minuto: se a mesma mensagem chegar duas vezes, a segunda inserĂ§ĂŁo Ă© rejeitada silenciosamente.

A consequĂŞncia importante: **a cadeia de FKs Ă© sempre consistente em reentregas**. Se `insertAge` passar mas `insertGender` falhar, o retry da mesma mensagem gera exatamente os mesmos UUIDs â€” o `age_id` passado para `target` serĂˇ idĂŞntico ao da primeira tentativa, sem criar linhas ĂłrfĂŁs.

### Campos NULLABLE mapeados para `*string`

As colunas `age_id`, `gender_id`, `social_class_id` e `target_id` sĂŁo NULLABLE no BigQuery. O serviĂ§o usa `*string` nos structs Go para que `InferSchema` (usado pelo `StructSaver`) gere o schema correto no lado cliente, alinhado com o que o Terraform provisionou.

### ValidaĂ§ĂŁo de chaves obrigatĂłrias antes de inserir

O campo `target` Ă© parseado como mapa (`map[string]float64`). Em Go, acessar uma chave inexistente em um map retorna o zero value â€” `0.0` para float64. Inserir `0.0` seria indistinguĂ­vel de um dado real de 0%, o que corromperia silenciosamente as anĂˇlises no Looker.

Por isso o serviĂ§o valida explicitamente todas as 16 chaves obrigatĂłrias antes de qualquer insert. Mensagem com chave ausente Ă© descartada com log de erro.

### NormalizaĂ§ĂŁo do campo `target`

A Claro Geodata envia o campo `target` como um dict Python serializado:


O serviĂ§o converte para JSON vĂˇlido via regex `'([^']*)'` â†’ `"$1"` antes de fazer o `json.Unmarshal`. A regex Ă© segura para o formato atual (chaves alfanumĂ©ricas, valores numĂ©ricos). ### Liveness probe via arquivo

O serviĂ§o escreve `/tmp/healthy` apĂłs processar a primeira mensagem com sucesso. O Kubernetes verifica esse arquivo a cada 30 segundos e reinicia o pod automaticamente se o consumer travar (ex: deadlock, conexĂŁo Kafka presa).

### Comportamento em falha de mensagem

O serviĂ§o usa `ReadMessage` do `kafka-go`, que faz auto-commit do offset apĂłs a leitura. Isso significa que **mensagens com erro sĂŁo puladas** â€” o consumer avanĂ§a o offset e a mensagem nĂŁo Ă© retentada automaticamente. O erro Ă© logado e o loop continua.
