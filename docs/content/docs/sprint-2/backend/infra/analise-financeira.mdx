# Análise Financeira — Infraestrutura em Nuvem

Este documento apresenta a análise de custos das três arquiteturas de nuvem propostas para o projeto Hiperconectividade para Cidades Inteligentes, desenvolvido pelo grupo Mídia in da House em parceria com a Eletromídia. 
## Por que uma Análise Financeira importa para o projeto?

Antes de construir qualquer coisa em nuvem, é fundamental entender *quanto custa rodar o que você está propondo*. Para um projeto acadêmico com impacto real como este, uma análise financeira cumpre três papéis críticos:

**Validação de viabilidade:** A Eletromídia precisa saber se a solução é sustentável para adotar em produção. Uma ferramenta tecnicamente brilhante que custa 10x mais do que o problema vale não será adotada.

**Apoio à decisão de arquitetura:** Ao comparar diferentes abordagens (gerenciada, serverless, Kubernetes), o grupo pode apresentar à empresa não só os trade-offs técnicos, mas também os financeiros — tornando a recomendação mais embasada e profissional.

**Transparência e credibilidade:** Projetar custos com base em dados reais (Google Cloud Pricing Calculator) e fórmulas documentadas demonstra rigor técnico e aumenta a confiança do parceiro no projeto.

Neste documento comparamos **três arquiteturas** para hospedar o Visualizador de Fluxo de Pessoas, todas calculadas com base nos dados reais de uso do projeto, operando em escala mínima mas com capacidade de crescimento irrestrita.

---

## Considerações Gerais

Antes de analisar os números, é importante registrar os critérios e premissas que nortearam todos os cálculos:

**1. Volume de dados baseado no fluxo real**  
Todas as métricas de armazenamento foram calculadas com base no volume real dos dados fornecidos pela Claro Geodata: **1 arquivo `fluxo_diario.csv` (9 MB) por dia**, totalizando **270 MB por mês**.

**2. Recursos computacionais baseados em requests do Kubernetes**  
Para arquiteturas que envolvem contêineres, os custos foram calculados com base nos *resource requests* de cada Pod — isto é, o mínimo de CPU e memória reservado por contêiner. A fórmula utilizada foi:

```text
Custo Mensal = [(CPU × Pcpu) + (RAM × Pram)] × Pods × 730h
```

Onde:
- `CPU` = requisição de CPU em vCPU
- `Pcpu` = preço por vCPU por hora no GKE Autopilot (**$0.0445/vCPU/h**)
- `RAM` = requisição de memória em GiB
- `Pram` = preço por GiB de memória por hora no GKE Autopilot (**$0.0049225/GiB/h**)
- `Pods` = número de réplicas em execução
- `730h` = total de horas em um mês (24h × ~30,4 dias)

> Os preços de CPU e memória do GKE Autopilot são os preços padrão (on-demand) da região **Iowa (us-central1)**, conforme tabela oficial do Google Cloud. Para a região **southamerica-east1 (São Paulo)** pode haver uma pequena variação de preço.

**3. Custos mínimos de operação**  
Todos os valores apresentados representam o **custo mínimo mensal**, considerando escala mínima (2 réplicas por serviço). Apesar disso, as três arquiteturas suportam escala elástica para qualquer volume de dados ou tráfego.

**4. Todos os serviços rodam com 2 réplicas 24/7**  
Em todas as três arquiteturas, cada serviço opera com **no mínimo 2 réplicas rodando continuamente**, 24 horas por dia, 7 dias por semana. Isso garante alta disponibilidade e elimina pontos únicos de falha. O número máximo de réplicas pode chegar a 15 por serviço durante picos de tráfego.

## Componentes Comuns a Todas as Arquiteturas

Independentemente da arquitetura escolhida, dois componentes de infraestrutura são sempre necessários:

### BigQuery BI Engine — $7,59/mês

Motor de análise in-memory que acelera consultas no BigQuery, viabilizando dashboards responsivos no Looker Studio mesmo com grandes volumes de dados. O custo é calculado com base nos **270 MB de dados reais** ingeridos por mês.

### Cloud Storage — $5,06/mês

Bucket para armazenamento e staging dos arquivos CSV recebidos da Claro Geodata. O valor de $5,06/mês considera uma multiplicação da carga real por 1.000 com **duplicação dos dados em duas regiões** (redundância regional), garantindo resiliência a falhas de zona.

> Em produção real (apenas 270 MB/mês), o custo do Cloud Storage seria praticamente **$0,00/mês**. O valor apresentado reflete um cenário de stress de dados para fins de estimativa conservadora.

**Subtotal fixo: $12,65/mês**

## Arquitetura 2 — Serverless Native

**Custo Total: $233,22/mês**  
🔗 [Ver cálculo no Google Cloud Pricing Calculator](https://cloud.google.com/products/calculator?hl=pt-BR&authuser=3&pageId=none&dl=CjhDaVE1Tm1NMU5HVTJPQzB6WVROa0xUUTVPR1l0T1RjMk15MHpZVEJpWm1Rek5UYzNZMllRQVE9PRALGiRCRDJBMUE1Ni0xMEU1LTQwMDctOTRGNS0yM0ZBRjNBQ0MzRkM)

### Descrição

Arquitetura baseada em **Cloud Run**, o serviço serverless de contêineres do Google. Os serviços de aplicação são deployados como contêineres stateless que escalam automaticamente do zero ao necessário, sem gerenciar instâncias ou clusters. O custo é proporcional ao uso real. No entanto, a camada de mensageria exige a manutenção de um **cluster Apache Kafka rodando em Compute Engines**, o que introduz um componente stateful e de gerenciamento manual na arquitetura — contrastando com a natureza serverless dos demais serviços.
### Componentes e Custos

| Serviço | Plataforma | Custo/mês |
|---------|------------|-----------|
| Cluster Apache Kafka | Compute Engine (3 instâncias) | ~141,72 |
| BigQuery Writer (2 réplicas) | Cloud Run | ~$19,71 |
| Cloud Storage Writer (2 réplicas) | Cloud Run | ~$19,71 |
| Kafka Geodata Producer (2 réplicas) | Cloud Run | ~$19,71 |
| Cloud Storage BFF (2 réplicas) | Cloud Run | ~$19,71 |
| BI Engine (BigQuery) | Analytics | $7,59 |
| Cloud Storage | Armazenamento | $5,06 |
| **Total** | | **$233,21** |


### Análise

Arquitetura com **ótimo equilíbrio entre custo e simplicidade operacional**. O Cloud Run elimina a necessidade de gerenciar clusters Kubernetes, mas ainda permite executar qualquer contêiner. O modelo de cobrança por request e por segundo de CPU/memória alocada garante que você pague apenas pelo que consome. É a opção **mais adequada para equipes de tamanho médio** que precisam de escalabilidade sem a complexidade do Kubernetes.

Um ponto de atenção importante nessa arquitetura, no entanto, é a **elasticidade do cluster Apache Kafka**. Como os Brokers Kafka rodam em **Compute Engines**, escalar a camada de mensageria horizontalmente não é trivial — diferente dos serviços Cloud Run, que escalam automaticamente. Para automatizar o escalonamento dos Brokers seria necessário implementar uma solução customizada, como **Terraform + Ansible** para provisionamento e configuração de novos nós, ou um **Autoscaling Group** com scripts de configuração pesados que garantam que cada novo Broker entre no cluster corretamente (registro no Zookeeper/KRaft, reassignment de partições, etc.). Essa limitação torna a manutenção mais trabalhosa e pode representar um gargalo em cenários de pico de ingestão de dados.

## Arquitetura 3 — Kubernetes-Native (GKE Autopilot)

**Custo Base (fixo): $12,65/mês + Custo Computacional: ~$164,43/mês**  
**Custo Total Estimado: ~$177,08/mês**

### Descrição

Arquitetura baseada em **GKE Autopilot**, o modo totalmente gerenciado do Google Kubernetes Engine. No Autopilot, você não gerencia nós — o Google provisiona a infraestrutura automaticamente com base nos *resource requests* dos Pods. Você paga apenas pelos recursos solicitados pelos Pods enquanto estão em execução.

### Preços GKE Autopilot (padrão on-demand)

| Recurso | Preço |
|---------|-------|
| vCPU (por hora) | **$0.0445** |
| Memória GiB (por hora) | **$0.0049225** |

> Fonte: [Tabela oficial de preços do GKE](https://cloud.google.com/kubernetes-engine/pricing) — região Iowa (us-central1), valores padrão sem desconto por compromisso.

| 2 - Cloud Storage Writer | 128m (0.128 vCPU) | 500Mi (0.488 GiB) | 2 / 15 |
| 3 - Kafka Geodata Producer | 128m (0.128 vCPU) | 500Mi (0.488 GiB) | 2 / 15 |

### Cálculo Detalhado por Serviço

Aplicando a fórmula `[(CPU × Pcpu) + (RAM × Pram)] × Pods × 730h`:

**Serviços 1, 2, 3 e 4** (idênticos):
```text
[(0.128 × $0.0445) + (0.4883 × $0.0049225)] × 2 × 730
= [$0.005696 + $0.002403] × 1.460
= $0.008099 × 1.460
= $11,82/mês por serviço
```

**Serviço 5:**
```text
[(0.500 × $0.0445) + (2.000 × $0.0049225)] × 5 × 730
= [$0.02225 + $0.009845] × 3.650
= $0.032095 × 3.650
= $117,15/mês
```

**PersistentVolume (10Gi — Standard Persistent Disk):**
```text
10 GiB × $0.04/GiB/mês = $0,40/mês
```

### Análise

Esta é a arquitetura **mais barata** em custo absoluto, sendo também a **mais complexa de operar**. O GKE Autopilot abstrai o gerenciamento de nós, mas ainda exige conhecimento de Kubernetes (manifests, deployments, services, HPA, PVCs). Em contrapartida, oferece o maior controle sobre configurações de rede, segurança e comportamento dos workloads. É a opção ideal para equipes com experiência em Kubernetes que buscam **máximo custo-benefício em longo prazo**, especialmente com descontos por compromisso de 1 ou 3 anos (que podem reduzir os custos de CPU e memória em até 46%).

## Recomendação

Para o escopo do projeto na **Inteli**, onde o foco é demonstrar viabilidade técnica com dados reais em escala mínima, a arquitetura **Kubernetes-Native com GKE Autopilot** oferece o melhor custo-benefício (~$177,08/mês), especialmente considerando que a equipe já está configurando a infraestrutura via Terraform (como documentado no módulo Terraform desta documentação).

Caso a Eletromídia opte por internalizar a solução sem equipe DevOps dedicada, a **Serverless Native (Cloud Run)** a $233/mês representa um caminho de menor fricção operacional com custo ainda controlado.
