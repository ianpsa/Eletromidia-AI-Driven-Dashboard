## Kafka

O cluster Kafka é gerenciado pelo operador **Strimzi** com modo **KRaft** (sem Zookeeper). Dois recursos definem o cluster: o `KafkaNodePool` (nós) e o `Kafka` (configuração do broker).

### Cluster (`template.yaml`)

**KafkaNodePool `dual-role`** — cada nó acumula os papéis de controller e broker:

| Campo                 | Valor                   |
| --------------------- | ----------------------- |
| Réplicas              | 3                       |
| Roles                 | `controller`, `broker`  |
| Storage               | JBOD, 10Gi por nó (PVC) |
| CPU request/limit     | `500m`                  |
| Memória request/limit | `2Gi`                   |

**Kafka `my-cluster`**:

| Configuração                               | Valor            |
| ------------------------------------------ | ---------------- |
| Listener plain (interno)                   | `:9092`, TLS off |
| Listener tls (interno)                     | `:9093`, TLS on  |
| `offsets.topic.replication.factor`         | `3`              |
| `transaction.state.log.replication.factor` | `3`              |
| `transaction.state.log.min.isr`            | `2`              |
| `default.replication.factor`               | `3`              |

O endereço interno do bootstrap para todos os serviços é:

```text
my-cluster-kafka-bootstrap.kafka.svc:9092
```

### Tópico (`geodata-topic.yaml`)

| Campo           | Valor              |
| --------------- | ------------------ |
| Nome            | `geodata`          |
| Namespace       | `kafka`            |
| Partições       | `3`                |
| Réplicas        | `2`                |
| `retention.ms`  | `86400000` (24h)   |
| `segment.bytes` | `1073741824` (1GB) |

### Makefile de debug

```bash
# Abre um producer interativo no tópico geodata
make -C src/kubernetes/kafka producer

# Consome todas as mensagens do início, com metadata
make -C src/kubernetes/kafka topic-logs

# Descreve partições e líderes do tópico
make -C src/kubernetes/kafka get-partitions
```

## Serviços

Todos os Deployments compartilham o mesmo padrão: variáveis de ambiente via ConfigMap e credenciais GCP via Secret montado como arquivo.

### bigquery-writer

Consome o tópico `geodata` e escreve nas 5 tabelas do BigQuery. Detalhes do serviço em [BigQuery Writer](/docs/sprint-2/backend/services/bigquery-writer).

**ConfigMap `bigquery-writer-cm`:**

| Variável                         | Valor                                       |
| -------------------------------- | ------------------------------------------- |
| `KAFKA_BROKERS`                  | `my-cluster-kafka-bootstrap.kafka.svc:9092` |
| `KAFKA_TOPIC`                    | `geodata`                                   |
| `KAFKA_GROUP_ID`                 | `bigquery-writer-group`                     |
| `KAFKA_MIN_BYTES`                | `1000`                                      |
| `KAFKA_MAX_BYTES`                | `10000000`                                  |
| `GCP_PROJECT_ID`                 | `midia-in-da-house`                         |
| `BQ_DATASET_ID`                  | `eletromidia_bq`                            |
| `GOOGLE_APPLICATION_CREDENTIALS` | `/secrets/key.json`                         |

**Deployment:**

| Campo                 | Valor                              |
| --------------------- | ---------------------------------- |
| Réplicas              | `1`                                |
| Estratégia            | `Recreate`                         |
| CPU request/limit     | `250m` / `500m`                    |
| Memória request/limit | `64Mi` / `128Mi`                   |
| `imagePullPolicy`     | `Never` (minikube local)           |
| Secret montado        | `gcp-sa-key` → `/secrets/key.json` |

A estratégia `Recreate` é usada localmente para evitar pico de memória com dois pods simultâneos. Em GKE com recursos suficientes, `RollingUpdate` pode ser adotado.

**Secret (somente local):**

```bash
# Gerar a chave via CLI do GCP
gcloud iam service-accounts keys create sa-key.json \
  --iam-account=bq-data-editor@midia-in-da-house.iam.gserviceaccount.com \
  --project=midia-in-da-house

# Criar o Secret no cluster
kubectl create secret generic gcp-sa-key \
  --from-file=key.json=./sa-key.json
```

Em produção (GKE), o pod autentica via Workload Identity e o Secret não é necessário. Ver `secret-README.md` para detalhes.

## Secrets por serviço

| Secret       | Serviços que usam                | O que contém                       |
| ------------ | -------------------------------- | ---------------------------------- |
| `gcp-sa-key` | `bigquery-writer`                | Chave SA com `bigquery.dataEditor` |
| `gcs-sa-key` | `bff-storage`, `bucket-consumer` | Chave SA com `storage.objectUser`  |

Ambos os Secrets são necessários apenas em **minikube local**. Em GKE Autopilot (produção), os pods autenticam via Workload Identity sem chaves de arquivo.

## Deploy completo (minikube)

```bash
# 1. Iniciar cluster
minikube start --driver=docker --memory=6144 --cpus=4 --disk-size=20g

# 2. Instalar Strimzi e provisionar Kafka
kubectl create namespace kafka
kubectl apply -f 'https://strimzi.io/install/latest?namespace=kafka' -n kafka
kubectl wait deployment/strimzi-cluster-operator -n kafka --for=condition=Available --timeout=300s
kubectl apply -f src/kubernetes/kafka/template.yaml -n kafka
kubectl apply -f src/kubernetes/kafka/geodata-topic.yaml -n kafka
kubectl wait kafka/my-cluster -n kafka --for=condition=Ready --timeout=600s

# 3. Apontar Docker para o daemon do minikube
eval $(minikube docker-env)

# 4. Criar Secrets (pedir chaves ao responsável GCP)
kubectl create secret generic gcp-sa-key --from-file=key.json=./sa-key.json
kubectl create secret generic gcs-sa-key --from-file=cs-api-key.json=./cs-api-key.json

# 5. Build e deploy dos serviços
docker build -t bigquery-writer:latest  src/services/bigquery-writer/
docker build -t bff-storage:latest      src/services/bff-storage/
docker build -t bucket-consumer:latest  src/services/bucket-consumer/

kubectl apply -f src/kubernetes/bigquery-writer/
kubectl apply -f src/kubernetes/bff-storage/
kubectl apply -f src/kubernetes/bucket-consumer/

# 6. Verificar pods
kubectl get pods
```
